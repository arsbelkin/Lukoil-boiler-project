# OPC UA (Unified Architecture)

## Зачем разработали OPC UA

Классический **OPC DA** получил огромное распространение, но имел принципиальный недостаток: он был построен на технологиях Microsoft (OLE, COM/DCOM) и поэтому работал **только под Windows**, плохо проходил через сети и не имел встроенной защиты.

Промышленность тем временем ушла вперёд: появились Linux, встраиваемые системы, облака, интернет вещей. Понадобился стандарт, независимый от операционной системы. Так появился **OPC UA** — *Unified Architecture*, «единая архитектура». Он закреплён международным стандартом **IEC 62541**.

> **OPC UA** — это современная спецификация OPC, которая **не** основана на технологии Microsoft COM и поэтому работает на любой платформе.

---

## Ключевая идея: узлы вместо плоского списка тегов

Главное отличие от OPC DA — модель данных. Вместо плоского списка тегов OPC UA использует **адресное пространство** из **узлов (nodes)**, связанных **ссылками (references)**. Это объектно-ориентированный подход: можно описывать сложные объекты реального мира.

Узлы бывают разных типов: **объекты**, **переменные** (те самые «теги»), **методы** (функции, которые можно вызвать), типы данных и др. Всё вместе образует дерево/граф:

```
Objects
└── Boiler                 ← объект
    ├── WaterLevel         ← переменная (тег)
    ├── OutputTemp         ← переменная (тег)
    ├── ValveHotIn         ← переменная (тег, доступна для записи)
    └── StartSimulation()  ← метод
```

Клиент (например, SCADA) может **просканировать** это дерево и сам узнать, какие объекты и теги есть на сервере — без заранее заданного «файла тегов».

---

## Две модели обмена данными

OPC UA поддерживает две модели связи:

1. **Клиент — сервер** — как в OPC DA: клиент подключается к серверу, читает/пишет значения, оформляет **подписки** (получает уведомления только при изменении). Хорошо работает для связи «точка — точка».
2. **Издатель — подписчик (PubSub)** — сервер публикует данные в сеть, а любой клиент может на них подписаться. Это связь «один ко многим» и «многие ко многим», удобная для IoT и большого числа устройств.

## Транспорт и безопасность

**Транспортные протоколы:**
- **OPC UA Binary over TCP** (`opc.tcp://...`) — быстрый бинарный протокол, основной вариант.
- **HTTPS** — для веб-интеграции.
- **PubSub over MQTT / UDP** — для промышленного интернета вещей.

**Встроенная безопасность** (чего не было у OPC DA):
- **Шифрование** передаваемых данных (TLS/SSL).
- **Аутентификация** через цифровые сертификаты **X.509**, а также логин/пароль.
- **Подпись сообщений** — защита от подмены.
- Гибкие политики доступа и роли пользователей.

---

## Информационные модели и реальное время

**Информационные модели (Companion Specifications).** Поверх базового стандарта отраслевые организации описывают типовые модели устройств (для упаковочных машин, роботов, приводов и т.д.). Это позволяет оборудованию разных производителей «понимать» друг друга «из коробки».

**OPC UA over TSN.** Для жёсткого реального времени OPC UA сочетают с технологией **TSN (Time-Sensitive Networking)** — расширением Ethernet, которое гарантирует низкие задержки и резервирование полосы. Это позволяет использовать OPC UA не только между SCADA и контроллерами, но и на полевом уровне — от датчиков к контроллерам и в облако.

Стандарт также делят по мощности устройства: **Nano / Micro / Embedded / Standard** — от крошечных датчиков до полноценных серверов.

## Плюсы и минусы

**Преимущества:**
- Кроссплатформенность (Windows, Linux, macOS, встраиваемые системы, облака).
- Встроенная безопасность (шифрование, сертификаты).
- Объектно-ориентированная модель данных с метаданными (единицы измерения, описания).
- Работает через NAT и брандмауэры без «прослоек».
- Поддержка не только данных, но и событий, команд и истории.

**Недостатки:**
- Сложнее в настройке, выше порог входа (сертификаты, сетевые настройки).
- Больше накладных расходов → для сверхбыстрых задач может уступать специализированным протоколам.
- Разные вендоры могут по-своему трактовать стандарт → редкие проблемы совместимости.
- Требовательнее к ресурсам (особенно с историей и сложными моделями).

## Сравнение OPC UA и OPC DA

| Характеристика     | OPC UA                          | OPC DA                     |
|--------------------|---------------------------------|----------------------------|
| **Платформа**      | Кроссплатформенный              | Только Windows             |
| **Основа**         | Открытые технологии, IEC 62541  | OLE / COM / DCOM           |
| **Безопасность**   | Шифрование, сертификаты X.509   | Нет (DCOM уязвим)          |
| **Модель данных**  | Объектно-ориентированная (узлы) | Плоская (список тегов)     |
| **Работа по сети** | Через NAT/брандмауэр            | Проблемы с DCOM            |
| **Модели связи**   | Клиент-сервер + PubSub          | Только клиент-сервер       |
| **История данных** | Встроенная поддержка            | Нет (нужен OPC HDA)        |

## Связь с нашим проектом (бойлер)

Наш проект построен именно на **OPC UA** — и наглядно показывает его идеи.

**Сервер** (`opc_server.py`) создаёт адресное пространство: объект `Boiler`, а на нём — переменные-теги.

```python
boiler = server.nodes.objects.add_object(idx, "Boiler")
boiler.add_variable(idx, "WaterLevel", 0.0)
boiler.add_variable(idx, "OutputTemp", 25.0)
node = boiler.add_variable(idx, "ValveHotIn", 0.5)
node.set_writable()   # клиент сможет менять положение клапана
```

**Клиенты** подключаются к серверу по адресу `opc.tcp://localhost:4840/...` — это тот самый **UA Binary over TCP**. У нас несколько клиентов, и все они общаются через один сервер:
- модель бойлера (`run_model.py`) — считает физику и пишет значения;
- SCADA-интерфейс (`app.py`) — показывает датчики и крутит клапаны;
- логгер (`history_logger.py`) — раз в 5 секунд читает и пишет в базу.

Здесь видны сразу несколько преимуществ OPC UA:
- **Древовидная структура** `Objects → Boiler → теги` вместо плоского списка (как было бы в OPC DA).
- **Доступ на запись** к отдельным тегам (`set_writable`) — оператор через SCADA меняет уставки клапанов.
- **Один сервер — много клиентов** одновременно, на любой ОС.

Именно поэтому для учебного проекта, имитирующего современную АСУ ТП, выбран OPC UA, а не устаревший OPC DA.

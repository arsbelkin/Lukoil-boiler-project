# Документация по интерфейсу (UI)

## Общие сведения

Веб-интерфейс — это SCADA-панель для мониторинга и управления моделью бойлера в реальном времени. Реализован на чистом HTML/CSS/JS без сторонних фреймворков.

**Файлы:**

| Файл | Назначение |
|---|---|
| `index.html` | Разметка страницы (карточки клапанов, SVG-схема бойлера, модальное окно) |
| `style.css` | Дизайн-система, стили компонентов, адаптивность |
| `script.js` | Опрос API, рендеринг данных в DOM/SVG, обработка модального окна |

Интерфейс работает по принципу **polling** — регулярно опрашивает REST API бэкенда и перерисовывает состояние. Пользователь не отправляет данные напрямую в DOM — все изменения проходят через API, а UI лишь отражает то, что вернул сервер (аналогично Unity-клиенту проекта).

## 1. Архитектура страницы

```
header (шапка, статус подключения)
└── main.dashboard (3-колоночная сетка)
    ├── section.valves-column.left   → карточки "Горячий вход", "Холодный вход"
    ├── section.boiler-section       → SVG-схема бойлера + баннер предупреждения
    └── section.valves-column.right  → карточка "Выход (слив)" + карточка с графиком
└── div.modal-overlay                → модальное окно редактирования значения
```

На узких экранах (`≤ 1023px`) сетка становится однoколоночной, карточки клапанов переходят в горизонтальный ряд с переносом (`flex-wrap`), а блок бойлера поднимается наверх (`order: -1`).

## 2. Шапка (`.header`)

- Заголовок приложения.
- Индикатор статуса соединения `#connectionStatus` с тремя состояниями:
  - `● Подключение...` — до первого ответа API;
  - `.status.connected` (зелёный) — `● Подключено`, последний запрос успешен;
  - `.status.error` (красный) — `● Ошибка соединения`, последний запрос завершился с ошибкой.

Статус обновляется функцией `setStatus(cls, text)` после каждого вызова `fetchData()` / `fetchGraph()`.

## 3. Карточки клапанов (`.valve-card`)

Всего три карточки: `hot` (горячий вход), `cold` (холодный вход), `out` (выход/слив). У каждой карточки атрибут `data-valve="valveHot|valveCold|valveOut"` — по нему скрипт находит нужное поле в объекте `state` для анимации ручки клапана.

### Структура карточки

1. **`.valve-visual`** — SVG-иконка крана (окружность-корпус + две линии-рукоятки, `.valve-handle`) и оверлей `.valve-percent` с текущим процентом открытия поверх иконки.
2. **`.valve-info`**
   - заголовок (название клапана);
   - `.info-row` — строки "Открытие: N%" и "Температура: N °C";
   - `.buttons` — кнопки редактирования (`.btn-edit`) с атрибутами:
     - `data-target` — какое поле состояния меняем (`valveHot`, `valveCold`, `valveOut`, `outputTemp`);
     - `data-type` — `percent` (0–100%) или `temp` (диапазон между `inputColdTemp` и `inputHotTemp`).

У входных клапанов доступна только кнопка изменения процента открытия (кнопка изменения температуры источника закомментирована в HTML — температуры горячего/холодного источника считаются заданными физическими параметрами, а не управляемыми через UI). У выходного клапана — две кнопки: процент открытия **и** целевая температура на выходе (уставка для ПИД-регулятора).

### Анимация ручки крана

В `render()` для каждой карточки вычисляется угол поворота рукоятки:

```js
const angle = (state[valveName] / 100) * 90; // 0° закрыт → 90° открыт
handles.forEach(h => h.style.transform = `rotate(${angle}deg)`);
```

Плавность обеспечивается CSS-переходом `transition: transform var(--transition)` на `.valve-handle` (0.5s, `cubic-bezier`).

### Карточка графика

Дополнительная карточка `.valve-card.graph` (правая колонка) не связана с клапаном — она отображает изображение графика, полученное с эндпоинта `GET /api/v1/graph` (см. раздел «Работа с API»). Картинка масштабируется через `object-fit: contain`, чтобы график всегда был виден целиком без искажений пропорций.

## 4. Схема бойлера (`.boiler-section`)

Центральный SVG-виджет (`viewBox="0 0 500 500"`) отрисовывает:

- **Трубы** (`.pipe`) — статичные серые линии входа/выхода.
- **Анимация потока** (`.pipe-flow`, классы `.hot-flow` / `.cold-flow` / `.out-flow`) — пунктирная линия поверх трубы, "бегущая" за счёт `stroke-dashoffset` (keyframe `flow`). Скорость и видимость потока зависят от процента открытия клапана — см. `setFlowSpeed()` ниже.
- **Бак** (`.tank`) — корпус бойлера со скруглёнными углами.
- **Уровень воды** (`#waterLevel`, класс `.water`) — прямоугольник внутри `clipPath`, высота и позиция `y` пересчитываются от `state.waterLevel` (0–100%) относительно высоты бака (290px) и его нижней границы (395px). Цвет воды динамически меняется в зависимости от температуры (см. `tempToColor()`).
- **Термометр** (`.thermometer`, `#thermoFill`, `.thermo-bulb`) — отдельный мини-индикатор температуры на выходе, диапазон 0–100 °C, плюс текстовые подписи текущей (`#tempText`) и целевой (`#tempTextTarget`) температуры.
- **Боковой индикатор уровня** (`#levelBar`, `.level-bar-fill`) — дублирующая вертикальная шкала уровня воды в процентах, с подписью `#levelText`.

### Баннер критического уровня (`#warningBanner`)

Появляется поверх схемы бойлера, если уровень воды приближается к переполнению:

- показывается при `waterLevel >= 98%` (`WARNING_THRESHOLD_ON`);
- скрывается при `waterLevel <= 95%` (`WARNING_THRESHOLD_OFF`);
- гистерезис (разные пороги на включение/выключение) сделан специально, чтобы баннер не "мигал" при колебаниях уровня около границы.

Визуально — красный градиентный баннер с пульсирующей тенью (`pulseWarning`) и трясущейся иконкой (`shake`), позиционирован абсолютно по центру над схемой бойлера.

## 5. Модальное окно редактирования (`#modalOverlay`)

Универсальное окно для изменения **любого** параметра — открывается по клику на кнопку `.btn-edit` любой карточки.

### Логика открытия

При клике на кнопку читаются `data-target` и `data-type`, вычисляются границы слайдера:

- **`type = "percent"`** → диапазон 0–100%;
- **`type = "temp"`** → диапазон от `state.inputColdTemp` до `state.inputHotTemp` (температура на выходе физически не может быть ниже холодного источника или выше горячего).

Заголовок окна формируется как `"Открытие: <название>"` или `"Температура: <название>"`, где название берётся из словаря `getDisplayName()`.

### Состав окна

- Заголовок + кнопка закрытия (`×`);
- слайдер `#modalSlider` с текущим значением, обновляющимся в реальном времени по событию `input`;
- отображение диапазона (мин/макс) под слайдером;
- кнопки **«Отмена»** (`#btnCancel`) и **«Применить»** (`#btnApply`).

### Закрытие окна

Окно можно закрыть тремя способами: кнопкой `×`, кнопкой «Отмена», кликом по затемнённой области вне окна (`modalOverlay`), либо клавишей `Esc`. Во всех случаях вызывается `closeModal()`, которая сбрасывает `modalContext` и убирает класс `.active` (окно скрывается через `display: none`, анимация `fadeIn`/`slideUp` проигрывается только при открытии).

### Применение изменений

По кнопке «Применить» вызывается `sendValue(target, value)` → `POST /api/v1/data` с телом `{ name, value }`. Локальное состояние **не обновляется оптимистично** — значение в интерфейсе появится только когда его подтвердит следующий опрос `GET /api/v1/data` (строки с `state[target] = value` намеренно закомментированы в коде). Это гарантирует, что UI всегда показывает то, что реально применено на бэкенде, а не то, что предположил клиент.

## 6. Работа с API (`script.js`)

| Метод | Эндпоинт | Когда вызывается |
|---|---|---|
| `GET` | `/api/v1/data` | Каждую секунду (`POLL_INTERVAL = 1000` мс) |
| `GET` | `/api/v1/graph` | Каждые 3 секунды (`POLL_INTERVAL_GRAPH = 3000` мс), возвращает PNG-изображение графика |
| `POST` | `/api/v1/data` | При нажатии «Применить» в модальном окне, тело `{ name: string, value: float }` |

### Состояние приложения (`state`)

Единый объект в памяти, хранящий последние известные значения: `inputHotTemp`, `inputColdTemp`, `outputTemp`, `waterLevel`, `valveHot`, `valveCold`, `valveOut`, `targetOutputTemp`. После каждого успешного `GET /api/v1/data` объект целиком перезаписывается (`Object.assign(state, data)`), после чего вызывается `render()`.

### Обработка ошибок

Если запрос завершился неуспешно (сетевая ошибка или HTTP-статус не `2xx`):
- статус в шапке переключается в `error`;
- для `sendValue()` дополнительно показывается `alert()` с текстом ошибки, чтобы пользователь понимал, что изменение не применилось.

### Вспомогательные функции рендеринга

- **`tempToColor(temp)` / `mixColors(c1, c2, k)`** — линейная интерполяция цвета воды: 0 °C → синий (`--color-cold`), 50 °C → фиолетовый (`--color-mixed`), 100 °C → красный (`--color-hot`).
- **`setFlowSpeed(selector, valvePercent)`** — если клапан закрыт (< 1%), анимация потока ставится на паузу и почти скрывается (`opacity: 0.2`); иначе поток идёт с прозрачностью `0.9`, а длительность анимации сокращается пропорционально открытию клапана (от 2s при малом открытии до 0.5s при 100%), создавая ощущение "чем сильнее открыт клапан, тем быстрее течёт вода".

## 7. Дизайн-система (`style.css`)

### Цветовая палитра (CSS-переменные в `:root`)

| Переменная | Значение | Назначение |
|---|---|---|
| `--color-hot` | `#e74c3c` | Горячая вода, предупреждения |
| `--color-cold` | `#3498db` | Холодная вода |
| `--color-mixed` | `#9b59b6` | Смешанная вода / выход |
| `--color-bg` | `#f5f5f5` | Фон страницы |
| `--color-accent` | `#2c3e50` | Акцентный тёмный (шапка, обводки, текст заголовков) |
| `--color-card` | `#ffffff` | Фон карточек |
| `--color-border` | `#dfe6e9` | Разделители |
| `--color-success` | `#27ae60` | Кнопка «Применить» |
| `--color-graph` | `#3f81e4` | Акцент карточки графика |

Дополнительно заданы тени (`--shadow-sm`, `--shadow-md`), скругление (`--radius: 12px`) и единая длительность плавных переходов (`--transition: 0.5s`), которая используется для уровня воды, термометра и цвета — чтобы вся анимация в SVG выглядела синхронно.

### Адаптивность

- **`≤ 1023px`** — сетка `.dashboard` становится однoколоночной; карточки клапанов выстраиваются в ряд с переносом; блок бойлера переносится наверх.
- **`≤ 767px`** — уменьшается заголовок, сокращаются отступы, карточки клапанов растягиваются на всю ширину, кнопки в `.buttons` складываются вертикально, баннер предупреждения уменьшает шрифт и паддинги.

## 8. Сопоставление UI ↔ полей API

| Поле в `state` / API | Где отображается в UI |
|---|---|
| `valveHot` | Карточка «Горячий вход»: процент, угол ручки, скорость потока |
| `valveCold` | Карточка «Холодный вход»: процент, угол ручки, скорость потока |
| `valveOut` | Карточка «Выход (слив)»: процент, угол ручки, скорость потока |
| `inputHotTemp` | Карточка «Горячий вход»: температура источника |
| `inputColdTemp` | Карточка «Холодный вход»: температура источника |
| `outputTemp` | Термометр, цвет воды в баке, карточка «Выход»: текущая температура |
| `targetOutputTemp` | Подпись целевой температуры под термометром (`#tempTextTarget`) |
| `waterLevel` | Уровень воды в баке, боковая шкала уровня, баннер критического уровня |

## 9. Возможные направления доработки

- Оптимистичное обновление `state` сразу после успешного `POST`, чтобы не ждать следующего цикла polling (сейчас закомментировано намеренно — компромисс между отзывчивостью и достоверностью данных).
- Замена polling на WebSocket/SSE для мгновенной синхронизации с Unity-клиентом и снижения нагрузки на API.
- Валидация вводимого значения температуры на клиенте до отправки (сейчас диапазон слайдера ограничивает ввод, но нет отдельной проверки перед `POST`).
- Локализация текстов баннера/подписей в отдельный словарь, если потребуется мультиязычность.
